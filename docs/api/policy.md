# API ポリシー（v1）

このドキュメントは、`/api/v1/banzuke.json` と `/api/v1/torikumi.json` の運用方針を定義します。

## 1. 更新スケジュール

API データは GitHub Actions の既存ワークフローに合わせて更新します。

- 日次更新（番付 + 取組）
  - 対象 workflow: `.github/workflows/daily-data-update.yml`
  - 実行タイミング: 毎日 19:00 JST（cron: `0 10 * * *`）
- 高頻度更新（取組のみ）
  - 対象 workflow: `.github/workflows/realtime-torikumi-update.yml`
  - 実行タイミング: 15:00-18:55 JST の間、5 分間隔（cron: `*/5 6-9 * * *`）

上記は定常運用の目安であり、障害・メンテナンス・データ取得元の都合で遅延または停止する場合があります。

## 2. 可用性目標

本 API は **ベストエフォート提供** です。

- 稼働率・応答時間・更新遅延に対する **SLA は提供しません**。
- 必要に応じて予告なく更新間隔・内容・配信方法を変更する場合があります。

## 3. 互換性ポリシー（v1）

`v1` では次の互換性を維持します。

- エンドポイントパス（`/api/v1/banzuke.json`, `/api/v1/torikumi.json`）
- 既存トップレベルキーの意味
- 既存フィールドの型（string / number / boolean / object / array）

次は互換性を壊さない変更として扱います。

- 新しいフィールドの追加
- 既存フィールドの値の追加（列挙値の追加を含む）
- 並び順の変更（意味論に影響しない範囲）

次は破壊的変更です。

- 既存フィールドの削除・改名
- 既存フィールドの型変更
- 既存フィールドの意味変更
- エンドポイントの削除・移動

破壊的変更は `v2` などの新バージョンで提供し、`v1` には原則として適用しません。

## 4. 廃止手順（Deprecation）

`v1` の機能・フィールド・エンドポイントを廃止する場合は、以下の手順で実施します。

1. GitHub Issues または README の告知欄で deprecation を告知
2. 告知から **最低 30 日間** の移行期間を設ける
3. `docs/api/changelog.md` に告知日・対象・予定日を記録
4. 移行期間終了後に廃止を実施

重大なセキュリティリスクや外部要因により、例外的に短縮または即時対応する場合があります。

## 5. 障害時の周知先

障害・更新遅延・仕様変更の周知は次で行います。

- GitHub Issues: <https://github.com/dai/o-sumo/issues>
- README の告知欄（日本語 / 英語）

運用上の重要な変更は `docs/api/changelog.md` にも記録します。
