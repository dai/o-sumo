# 力士プロフィール内製化 計画（コスト重視）

## 1. 背景と目的

現状、各力士のプロフィール導線は日本相撲協会プロフィール（外部サイト）へのリンクのみ。
本計画では、**外部リンクを残しつつ**、o-sumo 側で閲覧できる自前プロフィールページを追加する。

- 目的
  - サイト内回遊の改善（離脱を減らす）
  - API 利用者向けのプロフィール情報の一元化
  - 将来の比較・統計表示の拡張基盤を作る
- 制約
  - コスト重視
  - 更新頻度は「場所ごと（約2か月に1回）」を基本にする

## 2. 要件定義（MVP）

### 2.1 表示要件

- 力士一覧（`/api/v1/rikishi.json`）と個別（`/api/v1/rikishi/{id}.json`）に対応するページを作成
- 既存の「相撲協会プロフィール」リンクは明示して残す
- 自前プロフィールページに最低限表示する項目
  - 四股名
  - しこな（ローマ字）
  - 部屋
  - 番付
  - 出身地
  - 生年月日
  - 身長・体重
  - 初土俵
  - 得意技（取得できる場合）
  - 更新日時
  - 出典（日本相撲協会）

### 2.2 非機能要件

- 静的配信（低コスト）
- 既存ページの表示速度を悪化させない
- データ欠損時は「不明」で安全に表示
- 既存 URL と API を破壊しない

## 3. URL/画面設計

## 3.1 URL

- 新規ページ
  - `/rikishi`
  - `/rikishi/{id}`

## 3.2 導線

- 番付テーブル、取組ページ、検索結果（将来）から `/rikishi/{id}` へ遷移
- 個別ページ内に以下を併記
  - `相撲協会プロフィールを見る`（外部リンク）
  - `o-sumo API JSON`（`/api/v1/rikishi/{id}.json` へのリンク）

## 4. データ設計

## 4.1 追加ファイル（静的生成）

- `public/api/v1/rikishi/{id}.json` は既存を継続利用
- 追加を推奨
  - `public/api/v1/rikishi-index.json`（一覧表示・検索用の軽量索引）

## 4.2 取得・更新方針

- 既存 `scripts/update_sumo_data.py` にプロフィール抽出の正規化処理を追加
- 更新タイミング
  - 場所ごと（番付更新タイミング）
  - 手動実行コマンドを運用手順化

## 5. 実装ステップ（小さく刻む）

### Phase 1: データ基盤（1〜2日）

- `scripts/update_sumo_data.py` に以下を追加
  - 欠損時のデフォルト値
  - `updatedAt` 付与
  - `sourceUrl`（相撲協会プロフィールURL）を明示保持
- 生成物の差分確認手順を `README.md` に追記

### Phase 2: ページ追加（2〜3日）

- `app/rikishi/page.tsx`（一覧）
- `app/rikishi/[id]/page.tsx` 相当ルート（現行ルーティング規約に合わせて実装）
- 既存コンポーネントからプロフィールページへリンク追加

### Phase 3: 運用導入（0.5〜1日）

- 「場所ごと更新」Runbook を `docs/` に追加
- リリースチェックリスト整備
  - 型チェック
  - テスト
  - ビルド
  - 代表力士ページの目視確認

## 6. Cloudflare Pages / Workers の推奨構成

## 結論

**今回要件（コスト重視・隔月更新）なら、Cloudflare Pages 中心が最適。**

- 理由
  - プロフィールは更新頻度が低く、静的サイトと相性がよい
  - 既存運用が Pages 前提で、追加学習・移行コストが小さい
  - キャッシュ戦略が既存構成に乗る

## Workers paid の使いどころ（任意）

- 既存 Pages を維持しつつ、必要時のみ Workers を補助利用
  - Cron で隔月の自動データ更新ジョブ
  - 取得失敗時の通知
  - （将来）検索 API や差分配信

=＞ **推奨は「Pages 主体 + Workers 補助」のハイブリッド。**

## 7. コスト最適化ポイント

- まずは静的 JSON + 静的ページで実装
- 画像は当面外部リンク参照（自前ホスティングは後回し）
- 一覧用インデックス JSON を分離し、詳細 JSON の読み込みを遅延
- ビルド時に生成し、ランタイム計算を避ける

## 8. リスクと対策

- 取得元 HTML 変更でスクレイパーが壊れる
  - 対策: 失敗時に前回データを保持・警告を出す
- プロフィール項目の表記ゆれ
  - 対策: 正規化マップを `scripts/` 側で管理
- リンク切れ
  - 対策: CI でサンプル ID の外部リンクヘルスチェック（軽量）

## 9. 受け入れ基準（Definition of Done）

- `/rikishi` で一覧表示できる
- `/rikishi/{id}` でプロフィール表示できる
- 個別ページから相撲協会プロフィールへ遷移できる
- 既存ページ（番付・取組）で導線が1箇所以上追加される
- `npm run typecheck` / `npm test` / `npm run build` が通る
- 更新 Runbook に従って、場所単位更新が再現できる
